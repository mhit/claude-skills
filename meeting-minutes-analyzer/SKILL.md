# 会議議事録分析スキル

このスキルは、音声トランスクリプトから包括的で洞察に満ちた会議議事録を作成します。

## 概要

音声トランスクリプト(Whisperなどのツール出力)を入力として、以下を生成します:

1. **構造化された議事録** - 詳細な内容記録(箇条書きではなく、文脈重視)
2. **TODO・アクションアイテム** - 実行すべきタスクの抽出
3. **高度な分析レポート** - 会議内容に応じた多角的分析

### 対応形式

- **タイムスタンプあり/なし**: `[00:00 - 00:06]`形式、`0:03 | Me`形式、プレーンテキスト
- **話者名あり/なし**: 両方に対応
- **出力形式**: Word(.docx)優先、Googleドキュメントも対応可能

## ワークフロー

### ステップ1: トランスクリプトの確認と準備

ユーザーからトランスクリプトファイルを受け取ったら:

1. **ファイル確認**
   ```
   ファイルを確認させていただきます。
   [view tool でファイルの最初の50行を確認]
   ```

2. **形式の特定と説明**
   ```
   トランスクリプトの形式を確認しました。
   - 形式: [タイムスタンプあり/なし、話者名あり/なし]
   - 推定会議時間: [X分/時間]
   - 参加者: [判明している場合]
   
   これから議事録を作成していきます。
   ```

### ステップ2: 初期分析の実行

docxスキルを読み込んでから、トランスクリプト分析スクリプトを実行:

```bash
python3 scripts/analyze_transcript.py <transcript_file> > analysis_result.json
```

**重要**: このスクリプトは以下を抽出します:
- 固有名詞(人名、会社名、製品名)
- 会議タイプの自動分類
- 基本統計情報
- TODO候補
- キーワード

分析結果(JSON)を確認し、次のステップへ進みます。

### ステップ3: 固有名詞の確認 **[必須]**

分析結果から抽出された固有名詞を **必ずユーザーに確認** します:

```markdown
## 固有名詞の確認

トランスクリプトから以下の固有名詞を抽出しました。
正確な議事録作成のため、以下の表記が正しいかご確認ください:

### 人名
- もずみさん → 【正: 水海さん / 誤: そのまま / その他: ___】
- 伊藤さん
- 松下先生
- [他の人名]

### 会社名・組織名
- APEXコマース
- Amazon Japan
- [他の組織名]

### 製品・ブランド名
- スノーフォックス
- フィービー
- アダマスオクター
- [他の製品名]

### その他の固有名詞
- [場所名、プロジェクト名など]

**ご回答方法**: 
- 正しい表記を【正:】の後に記入
- そのままで良い場合は「OK」または「そのまま」とご記入
- 不明な場合は「不明」とご記入ください

この確認後、正確な議事録を作成いたします。
```

**ユーザーからの回答を待ってから次のステップへ進むこと。**

### ステップ4: 会議タイプの判定と分析フレームワークの選択

スクリプトの分析結果(`meeting_classification`)を基に会議タイプを判定:

| 会議タイプ | 特徴 | 優先フレームワーク |
|-----------|------|-------------------|
| strategic | 戦略、ビジョン、方針の議論 | SWOT分析、意思決定ツリー、キーワード分析 |
| brainstorming | アイデア出し、提案 | テキストマイニング、キーワード分析、感情分析 |
| status_update | 進捗報告、状況共有 | プロセスマイニング、貢献度分析 |
| problem_solving | 課題解決、対策検討 | ロジックツリー、SWOT分析、意思決定ツリー |
| client_meeting | 顧客・取引先との打ち合わせ | 感情分析、意思決定ツリー |
| interview | 採用面接 | 感情分析、キーワード分析(通常分析控えめ) |
| product_discussion | 製品・開発の議論 | 意思決定ツリー、SWOT分析、ロジックツリー |

**フレームワーク選択の基準**:
- 会議タイプの`scores`が高い上位1-2カテゴリを採用
- 最低1つ、最大3つのフレームワークを選択
- 詳細は`references/analysis_frameworks.md`を参照

### ステップ5: 分析フレームワークの適用

選択したフレームワークに基づいて詳細分析を実施します。

**必須手順**:
1. `references/analysis_frameworks.md`を読み込む
2. 選択したフレームワークのセクションを参照
3. ガイドラインに従って分析を実行

例: SWOT分析を適用する場合
```
[file_read: references/analysis_frameworks.md の SWOT分析セクション]
[トランスクリプトからS・W・O・Tに該当する発言を抽出]
[ガイドラインに従ってマークダウン形式で整理]
```

### ステップ6: 議事録の作成(Word形式)

**重要**: 必ず`/mnt/skills/public/docx/SKILL.md`を最初に読み込んでください。

```bash
[file_read: /mnt/skills/public/docx/SKILL.md]
```

議事録はWord文書として作成します。以下の構成を推奨:

#### 議事録の構成

1. **表紙・基本情報**
   ```
   会議議事録
   
   会議名: [会議のタイトル/テーマ]
   日時: [日付と時間]
   参加者: [参加者リスト]
   記録者: [AIによる自動生成]
   ```

2. **エグゼクティブサマリー** (1-2段落)
   - 会議の目的と主要な結論を簡潔に記載
   - 重要な決定事項を3-5点リストアップ

3. **詳細議事録**
   
   **重要ガイドライン**:
   - ❌ 単純な箇条書きは避ける
   - ✅ 段落形式で文脈と背景を含めて記述
   - ✅ 「誰が」「なぜ」「どのように」を明確に
   - ✅ 議論の流れと関連性を保持
   
   **推奨構成**:
   ```markdown
   ### 1. [トピック名]
   
   [2-3段落の詳細な記述]
   
   **主な論点**:
   - [ポイント1の詳細説明]
   - [ポイント2の詳細説明]
   
   **決定事項**: [決定内容と理由]
   
   **発言者の視点**:
   - [Aさんの意見と根拠]
   - [Bさんの懸念点]
   ```

4. **TODO・アクションアイテム**
   
   | タスク | 担当者 | 期限 | 優先度 | 備考 |
   |--------|--------|------|--------|------|
   | [具体的なタスク] | [名前] | [日付] | 高/中/低 | [補足] |

5. **決定事項サマリー**
   - 会議で決定したことを明確にリスト化

6. **次回までのアクション**
   - 次回会議までに完了すべき事項

7. **高度な分析レポート** (選択されたフレームワークに基づく)
   
   各分析フレームワークごとにセクションを設ける:
   - 意思決定ツリー分析
   - SWOT分析
   - 感情分析
   - キーワード・トピックトレンド分析
   - 貢献度・影響力分析
   - ロジックツリー分析
   - プロセスマイニング
   - テキストマイニング
   
   **注意**: 会議タイプに応じて1-3個のフレームワークを選択適用

8. **付録**
   - 用語集(必要に応じて)
   - 参考資料リスト

#### フォーマッティング

- **見出し**: 階層構造を明確に(Heading 1, 2, 3を使用)
- **強調**: 重要な決定事項やキーワードは **太字**
- **表**: アクションアイテムや比較データ
- **箇条書き**: 最小限に留める(詳細は段落で)

#### ファイル作成

```python
# docxスキルの指示に従ってWord文書を作成
# /mnt/user-data/outputs/ に保存
```

### ステップ7: Googleドキュメント版の作成(オプション)

ユーザーがGoogleドキュメント形式も希望する場合:

1. Word版の内容をベースに作成
2. Markdown形式で一時ファイルを作成
3. Google Drive にアップロードできる形式に変換

**注意**: Word版が優先です。Googleドキュメントは明示的に要求された場合のみ作成。

### ステップ8: 最終確認と納品

1. **ファイル確認**
   - Word文書が `/mnt/user-data/outputs/` に正しく作成されているか
   - ファイルサイズと内容の妥当性

2. **ユーザーへの提示**
   ```markdown
   会議議事録を作成しました。
   
   ## 📄 議事録ファイル
   
   [議事録をダウンロード](computer:///mnt/user-data/outputs/meeting_minutes_YYYYMMDD.docx)
   
   ## 📊 分析サマリー
   
   - **会議タイプ**: [タイプ]
   - **参加者数**: [X名]
   - **主要トピック**: [トピック1, トピック2, トピック3]
   - **決定事項**: [X件]
   - **アクションアイテム**: [X件]
   - **適用した分析フレームワーク**: [フレームワーク名]
   
   ## ✨ 分析のハイライト
   
   [主要な洞察を2-3点、簡潔に記載]
   
   ご確認ください。修正や追加分析のご希望があれば、お気軽にお申し付けください。
   ```

## 実装ガイドライン

### 品質基準

1. **正確性**
   - 固有名詞は必ずユーザー確認後に使用
   - トランスクリプトの内容を正確に反映
   - 推測や創作は行わない

2. **詳細度**
   - 「中身を重視」するため、単なる箇条書きではなく段落形式で記述
   - 背景や文脈を含める
   - 議論の流れを論理的に再構成

3. **分析の深さ**
   - 選択したフレームワークを適切に適用
   - 単なる要約ではなく、洞察を提供
   - 具体的なデータや引用を含める

4. **読みやすさ**
   - 明確な構造(見出し、段落、表)
   - 適切なフォーマッティング
   - 専門用語には説明を付ける(必要に応じて)

### トラブルシューティング

**トランスクリプトの品質が低い場合**:
- 可能な範囲で内容を推測・補完
- 不明瞭な部分は [不明瞭] と明記
- ユーザーに確認を求める

**話者名が不明な場合**:
- 「話者A」「話者B」などの仮名を使用
- 発言内容から推測できる場合は注記

**会議タイプが判定できない場合**:
- 一般的な議事録形式を採用
- 基本的な分析(キーワード分析、アクション抽出)のみ実施

## 重要な注意事項

1. **必ず固有名詞の確認を行う** - これをスキップしないこと
2. **docxスキルを読み込んでからWord文書を作成** - Word作成の品質担保
3. **分析は会議内容に応じて柔軟に** - すべてのフレームワークを適用しない
4. **箇条書きより段落を優先** - 「中身を重視」の原則
5. **Word優先、Googleドキュメントは補助** - 出力形式の優先順位を守る

このワークフローに従うことで、単なる記録ではなく、組織の意思決定や生産性向上に貢献する価値ある議事録を作成できます。
